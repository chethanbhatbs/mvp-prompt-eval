Verdict

v2_credit_cap cuts ECU 22–56% and is faster on every prompt — the conversion win is real. BUT the saving is partly because it builds less: on the complex Task-Management prompt (P1) v2_credit_cap shipped thin Phase-1 slices (code quality 4/10 "Weak", ~2/10 spec coverage) vs Normal Prompt's fuller builds (6–7/10). On P2 (a naturally small app) v2_credit_cap matches Normal Prompt quality (8/10) at lower cost — a clean win. On P3 quality is comparable (~6/10 both) at ~56% lower cost — a win. Net: v2_credit_cap is the right default for getting users to a working result inside the 10-credit cap, but on large prompts it under-builds — the deferred scope must actually get built in phase 2, or users receive a thin product.

Normal Prompt = gflash environment (normal prompt). v2_credit_cap = long-prompt environment (v2 prompt with credit cap). ECU/latency from credit_ledger + trajectories; code quality = static review 1–10. P4 (one arm only) and P5 (not run) excluded.
